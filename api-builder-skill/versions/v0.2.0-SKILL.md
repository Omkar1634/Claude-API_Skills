# API Builder Skill

## Overview

API Builder is a Claude Skill that scaffolds a backend API project through an interactive step-by-step wizard artifact. It collects all project information via clickable option cards, shows a confirmation summary, then calls the Claude API to generate a complete project — with individual file view/download options.

---

## Trigger Phrases

Activate this skill when the user says any of the following (or close variations):

- "build an API"
- "create a backend"
- "scaffold my API"
- "help me build a REST API"
- "I want to create an API"
- "set up a backend project"

---

## Core Rules (Always Enforced)

1. **Launch the artifact immediately** — do not ask questions conversationally. Render the wizard artifact on trigger.
2. **Never assume the framework** — framework options are shown only after language is selected.
3. **No experience level question** — removed in v0.2.0.
4. **Always ask permission before reading existing code** — for existing project flow only.
5. **Show folder structure first** — before any file content is shown.
6. **Files shown with View and Download buttons** — never dump full code into chat.
7. **Generated code uses real entity names** — never generic placeholders like MyModel or YourEntity.
8. **Confirm summary before generating** — full recap shown at Step 10, generation only starts on confirmation.

---

## Delivery Method

When this skill activates, launch the **API Builder Wizard** — an interactive Claude Artifact that collects all project information through a step-by-step UI with clickable option cards. Do NOT ask questions conversationally. Instead render the artifact immediately.

The artifact handles:
- All question steps with selectable option cards
- Pre-selected feature suggestions based on context
- Full configuration summary before generation
- Claude API call to generate the complete project
- Folder structure tree displayed before files
- View and Download buttons per file — no code dumped in chat
- Download all option for the complete project

After the artifact completes, remain available for follow-up requests in chat (add endpoint, generate tests, etc.)

## Wizard Steps

### Step 1 — Project type
Single select: From scratch / Mid-project / Existing API

### Step 2 — Project description
Free text — what does it do, who is it for

### Step 3 — Entities
Free text — main resources (users, products, orders etc.) — used as real names in generated code

### Step 4 — API consumers
Multi-select: Web frontend / Mobile app / Other services / Third-party clients

### Step 5 — Language
Single select: Python / JavaScript / TypeScript / C# / Java / Go / PHP / Ruby

### Step 6 — Framework
Single select — options shown based on chosen language:

| Language | Options |
|---|---|
| Python | FastAPI, Django REST, Flask |
| JavaScript | Express, Fastify, NestJS |
| TypeScript | NestJS, Fastify, Express |
| C# | ASP.NET Core Web API, Minimal API |
| Java | Spring Boot, Quarkus |
| Go | Gin, Echo, Fiber |
| PHP | Laravel, Slim |
| Ruby | Rails API, Sinatra |

### Step 7 — Database
Single select: SQLite / PostgreSQL / MySQL / MongoDB

### Step 8 — Auth
Single select: JWT / API Key / OAuth2 / Session / None

### Step 9 — Features
Multi-select with pre-suggestions applied automatically:

| If... | Then pre-select... |
|---|---|
| Any consumer selected | CORS Config |
| Always | OpenAPI / Swagger, Structured Logging |
| Web or mobile consumer | Rate Limiting |
| List-based entities (products, posts, orders) | Pagination |
| Docker mentioned | Docker |

Full feature options:
- OpenAPI / Swagger
- CORS Config
- Structured Logging
- Rate Limiting
- Pagination
- Caching
- Unit Tests
- Docker

### Step 10 — Confirm summary
Full recap of all selections. User reviews before generation starts.

### Step 11 — Generate
Claude API call generates complete project. Output:
1. Folder structure tree shown first
2. File list with View and Download buttons per file
3. Download all button for complete project

## Generated Project Structure (FastAPI + SQLite example)

```
my-api/
├── app/
│   ├── __init__.py
│   ├── main.py              — FastAPI app entry, mounts routers, registers middleware
│   ├── config.py            — Settings via pydantic-settings, reads .env
│   ├── dependencies.py      — Shared DI: db session, current user
│   ├── api/
│   │   ├── __init__.py
│   │   └── v1/
│   │       ├── __init__.py
│   │       ├── router.py    — Aggregates all v1 sub-routers
│   │       └── endpoints/   — One file per resource (auth.py, users.py, products.py)
│   ├── core/
│   │   ├── security.py      — JWT encode/decode, password hashing
│   │   ├── rate_limit.py    — slowapi integration
│   │   ├── exceptions.py    — Global HTTP exception handlers
│   │   ├── logging.py       — Structured logging setup
│   │   └── pagination.py    — Reusable page/limit query params
│   ├── db/
│   │   ├── session.py       — SQLAlchemy engine + SessionLocal
│   │   ├── base.py          — DeclarativeBase all models inherit
│   │   └── migrations/      — Alembic migration files
│   ├── models/              — SQLAlchemy ORM table definitions (one file per entity)
│   ├── schemas/             — Pydantic request/response models (one file per entity)
│   │   └── common.py        — PaginatedResponse, ErrorDetail
│   └── services/            — Business logic layer (one file per entity)
├── tests/
│   ├── conftest.py
│   ├── unit/
│   └── integration/
├── .env.example
├── requirements.txt
├── requirements-dev.txt
├── alembic.ini
├── Dockerfile               — if Docker selected
├── .gitignore
└── README.md
```

Each generated file must be:
- Complete and immediately runnable
- Named after real entities from user input
- Free of TODO comments and placeholder values

## Phase 1B — Existing Project Flow

If user selects "Existing API" in Step 1:

1. Ask permission to read the code:
   > "Before I look at anything, do I have your permission to read your existing code? You can either share the files directly or paste the relevant code here."

2. Wait for user to share. Do not reference any code until they do.

3. Show an implementation plan:
   > "Here's what I'd suggest adding to your project: [plan summary]. Does this look right before I proceed?"

4. Ask delivery preference:
   > "Would you prefer I edit the files directly, or would you like me to show you the code blocks to copy in yourself?"

5. Wait for confirmation before making any changes.

---

## Phase 1C — Mid-project Flow

If user selects "Mid-project" in Step 1:

Acknowledge what they have, ask what the API needs to do, then proceed through the wizard using their existing context. Reference their frontend/DB when making suggestions (CORS, data shape matching etc.) Do NOT ask for existing backend code since there is none.

---

## Follow-Up Loop

After the artifact completes generation, offer in chat:

1. Add a new endpoint
2. Generate tests
3. Show the OpenAPI spec
4. Add Docker support
5. Something else

Handle each, then return to the menu until the user is done.
